# 分析型内部原型设计规格

## 1. 目标

本阶段的目标不是直接做对外产品，也不是优先追求论文叙事，而是构建一套可用于内部技术验证的分析型原型。该原型应当能够基于固定复盘样本集，对当前车辆与车牌识别链路进行结构化诊断，明确系统瓶颈究竟主要来自车牌检测、车牌裁剪与前处理、OCR 识别，还是模块间匹配与结果组织。

第一阶段的核心价值是“看清问题”，而不是“把界面做漂亮”。只要这套原型能够稳定地产出批量诊断报告，并支持对失败样本进行逐张复盘，它就已经满足第一阶段设计目标。

## 2. 产品定位

本原型定位为内部验证工具，主要服务于研发者本人或内部协作者。它不面向普通用户，也不假设外部用户会直接上传图片使用。其使用方式是：在服务器侧使用既有模型和配置运行固定复盘集推理，然后在本地读取结果、生成批量报告，并基于报告判断下一轮应该优先优化哪个模块。

## 3. 主场景

第一阶段只关注单张图片识别链路，不把视频、实时摄像头或批量对外上传作为核心目标。

单张图片链路必须覆盖以下步骤：

1. 车辆检测
2. 车牌检测
3. 车牌与车辆匹配
4. 车牌裁剪与矫正
5. OCR 原始输出与归一化输出
6. 结果导出与错误归因

## 4. 主输出

第一阶段的主输出是批量诊断报告，而不是网页式在线交互界面。报告需要支持离线查看、版本对比和失败样本复盘。

报告建议包含以下结构：

1. 总览页
   说明本次复盘的样本数量、模型版本、配置文件、总体字符级准确率、整牌完全正确率、空结果率、以及“检测到牌但识别失败”的比例。
2. 分类统计页
   按固定样本类别展示统计结果，至少区分容易样本、模糊样本、倾斜样本和明显失败样本。
3. 失败样本页
   汇总最典型的失败样本，展示原图、车牌框、裁剪图、矫正图、最终文本、真值文本、错误类型和关键置信度。
4. 明细表
   对每张样本给出结构化记录，便于筛选和后续分析。
5. 产物索引
   记录本次运行使用的输入、模型、配置、报告文件和中间产物目录，保证结果可追溯。

## 5. 输入定义

第一阶段原型固定依赖以下四类输入：

### 5.1 固定复盘样本清单

不得每次随机抽样。必须形成一份固定样本清单，保证不同模型版本之间结果可横向比较。

第一版采用分层复盘集思路，样本来源于现有数据集和烟测样本，按以下类别组织：

1. 容易样本
2. 模糊样本
3. 倾斜样本
4. 明显失败样本

后续如果有自采真实图片，再单独扩展一组“真实场景样本”，但不作为第一阶段前置条件。

### 5.2 模型版本

每次报告都必须显式记录以下版本信息：

1. 车牌检测模型路径或版本标识
2. OCR 模型路径或版本标识
3. 运行配置文件路径
4. 推理时间戳

### 5.3 中间产物输出

每张图片需要固定输出下列中间结果，供后续排查使用：

1. 原图
2. 标注图
3. 车牌裁剪图
4. 矫正后车牌图
5. OCR 原始文本
6. OCR 归一化文本
7. 关键置信度

### 5.4 评估规则

第一阶段仅保留最能指导产品判断的指标，不引入复杂但难解释的指标体系。首批指标定义如下：

1. OCR 字符级准确率
2. 整牌完全正确率
3. OCR 空结果率
4. 检测到牌但识别失败率

## 6. 错误归因框架

报告必须能把失败样本归入可操作的问题类型，而不是只展示“错了”。第一版错误归因建议至少支持以下类别：

1. 未检测到车牌
2. 车牌框偏差过大，导致裁剪质量差
3. 车牌检测到，但未匹配到车辆
4. OCR 返回空结果
5. OCR 返回错误文本
6. OCR 返回重复字符或明显异常文本

这套归因框架的目的是让下一轮决策有方向。例如，如果大量错误属于“裁剪质量差”，则后续优先投入前处理；如果主要属于“OCR 返回异常文本”，则优先投入 OCR 数据与模型。

## 7. 运行架构

第一阶段采用“服务器推理，本地查看与复盘”的模式。

1. 服务器侧负责：
   - 读取固定复盘样本清单
   - 运行现有检测与 OCR 模型
   - 导出结构化结果和中间产物
2. 本地侧负责：
   - 拉取或读取导出结果
   - 汇总统计指标
   - 生成诊断报告
   - 供人工复盘查看

采用这种模式的原因是：当前最终模型和演示配置主要存在于服务器环境，若强行把第一阶段做成本地完整复现，会把精力浪费在环境问题而不是诊断能力上。

## 8. 非目标

第一阶段明确不做以下内容：

1. 实时摄像头识别
2. 视频主流程优化
3. 外部用户可直接使用的产品前端
4. 大规模重训或全面模型重构
5. 自动化部署与权限系统
6. 对外上线能力

## 9. 验收标准

第一阶段完成的判断标准不是模型是否显著提升，而是分析能力是否成型。验收标准如下：

1. 存在一份固定复盘样本清单
2. 服务器可基于该样本清单稳定导出推理结果和中间产物
3. 本地可基于导出结果生成批量诊断报告
4. 报告能够展示总览、分类统计、失败样本和明细表
5. 至少能够对主要失败样本做出结构化错误归因
6. 报告结论能够明确指出下一阶段优先投入的模块

## 10. 风险与后续决策

第一阶段最主要的风险是：报告做出来后，发现错误来源高度混杂，仍然难以确定瓶颈。为降低这一风险，第一版必须尽量保留中间产物和版本信息，而不是只保留最终文本。

如果第一阶段执行完成，后续路线将根据报告结论二选一：

1. 若瓶颈主要在 OCR，则进入 OCR 数据与模型强化阶段；
2. 若瓶颈主要在检测与裁剪，则进入检测框质量与前处理强化阶段。

## 11. 当前已确认前提

1. 产品长期目标是“后续能继续做成产品”，但当前阶段先做内部原型。
2. 第一阶段不是外部产品，而是分析型内部原型。
3. 第一阶段以服务器结果为主，本地负责查看和复盘。
4. 当前没有自采真实图片，第一版样本集基于现有数据和烟测样本构建。
5. 第一阶段主输出是批量诊断报告。
6. 当前范围已明确排除实时视频、对外前端和大规模重训。
