# OCR 小样本评估设计

**日期：** 2026-04-12

## 目标

补一个面向论文与实验验证的 OCR 小样本评估脚本，在同一批 CCPD 样本上对比“优化前 OCR”与“优化后 OCR”的表现，并输出可复核的结果文件。

本次评估只回答一个问题：当前这轮 OCR 稳定化改动，是否在小样本上带来了更好的整牌结果、更高的字符级准确率，或者更保守的空结果行为。

## 范围

### 包含

- 使用 `D:\plate_project\CCPD2019` 作为样本来源
- 使用 CCPD 文件名中的标注框作为真值裁牌依据
- 使用 CCPD 文件名中的车牌编码作为真值文本来源
- 在同一批样本上比较两种 OCR 模式
- 输出主指标、辅助指标、补充指标
- 输出逐样本明细，便于论文截图和人工复核

### 不包含

- 不评估车牌检测器效果
- 不接入 YOLO 预测框进行端到端比较
- 不连接云服务器
- 不在本阶段修改生产推理主链
- 不夸大结论，所有表述以真实评估结果为准

## 为什么采用这个方案

这次要验证的是 OCR 稳定化效果，而不是检测器效果。若把检测器一起纳入评估，检测框误差会混入 OCR 指标，导致“优化前 vs 优化后”的归因不清晰。

因此，评估应直接使用 CCPD 真值框裁牌，只让变量集中在 OCR 行为本身。这样既能保持实验规模可控，也更适合作为论文中的“小样本补充验证”。

## 推荐方案

采用“分场景分层抽样 + 真值裁牌 + 双模式 OCR 对比”的方案。

核心思路如下：

1. 从 CCPD 测试集按子集分层抽样，而不是全随机抽样
2. 使用文件名中的真值框直接裁出车牌区域
3. 在同一张裁牌图上分别运行“优化前 OCR”和“优化后 OCR”
4. 使用统一的评估口径计算整牌完全正确率、字符级准确率、无结果样本数
5. 同时保留逐样本明细，便于人工检查异常样本

## 样本来源与抽样策略

### 数据来源

- 数据集根目录：`D:\plate_project\CCPD2019`
- 默认划分文件：`D:\plate_project\CCPD2019\splits\test.txt`

### 抽样对象

默认只从有车牌的子集中抽样，不纳入 `ccpd_np`。

默认优先覆盖以下子集：

- `ccpd_base`
- `ccpd_blur`
- `ccpd_db`
- `ccpd_rotate`
- `ccpd_tilt`
- `ccpd_weather`
- `ccpd_challenge`

若某个子集在当前划分文件中没有足够样本，则脚本按实际可用数量抽取，并在汇总中记录。

### 抽样方式

- 先按子集分组
- 再在每个子集内用固定随机种子抽样
- 保持样本顺序可复现

默认建议：

- `split = test`
- `seed = 42`
- `per_subset = 10`

按上述默认值，总样本量目标为 `70`。这个规模足够形成基础对比，同时仍属于“小样本评估”，本地运行成本也可控。

## 真值解析与评估口径

### 真值来源

脚本需要基于 `src/car_system/data/ccpd.py` 现有的 `parse_ccpd_path` 能力，补一层“CCPD 索引到车牌文本”的解码逻辑，从文件名恢复车牌真值。

每个样本应至少保留两种真值表示：

- `gt_full_text`：完整车牌文本
- `gt_eval_text`：用于本次评估比较的规范化真值

### 为什么需要 `gt_eval_text`

当前系统的 OCR 输出口径并不是“原始完整中文车牌串”。现有 `src/car_system/ocr/plate_ocr.py` 中的文本规范化只保留英文字母和数字，中文省份简称不会进入最终输出。

因此，如果直接拿完整中文车牌串与当前系统输出比较，整牌完全正确率会因为口径不一致而失真。

### `gt_eval_text` 定义

`gt_eval_text` 采用与当前系统输出一致的比较口径：

1. 先将完整真值转为大写
2. 仅保留 ASCII 字母与数字
3. 不对真值做额外的易混字符替换

示例：

- 完整真值：`京A12345`
- 评估真值：`A12345`

这个口径的意义不是宣称系统已经识别完整中文牌照，而是保证“优化前 vs 优化后”比较建立在当前系统真实输出能力之上。

## 对比模式定义

### 模式一：优化前 OCR

优化前模式用于近似复现本轮稳定化之前的 OCR 行为，作为评估基线。

该模式应满足：

1. 使用 CCPD 真值框直接裁牌，不做额外外扩
2. 保持现有 `rectify_plate` 钩子，但不增加新处理
3. 仅对原始裁牌图进行一次 OCR 识别
4. 不生成多候选图
5. 不做候选结果排序
6. 不做本轮新增的易混字符修正
7. 不做“过短脏结果丢弃”

该模式的目标不是完全回滚整个仓库，而是在评估脚本中提供一个可重复、可解释的“旧行为参考实现”。

### 模式二：优化后 OCR

优化后模式应尽量复用当前正式实现，对齐现有稳定化逻辑。

该模式应满足：

1. 按当前 runner 逻辑对车牌框做小幅外扩后再裁牌
2. 使用现有 `rectify_plate`
3. 复用当前 `PaddlePlateOCR.recognize`
4. 包含多候选图、轻量清洗、候选排序、易混字符修正和过短结果过滤

这样可以保证评估结果直接对应当前系统的真实行为，而不是另写一套口径相近但实现不同的“伪优化版”。

## 指标定义

### 主指标：整牌完全正确率

定义为：

`pred_text == gt_eval_text`

统计方式：

- 分模式统计完全正确样本数
- 完全正确样本数 / 有效样本总数

这是论文中最容易解释、也最核心的指标。

### 辅助指标：字符级准确率

定义为：

- 对每个样本按字符位置逐位比较
- 单样本正确字符数 / 该样本 `gt_eval_text` 长度
- 汇总时使用“全部样本正确字符总数 / 全部样本真值字符总数”

规则：

- 预测为空时，单样本字符正确数记为 `0`
- 预测长度超过真值长度时，超出部分不计正确
- 预测长度不足时，缺失部分计错

该指标用于反映“虽然整牌没全对，但是否更接近真值”。

### 补充指标：无结果样本数

定义为：

- 预测结果为 `None`
- 或最终评估串为空

该指标用于刻画优化后更保守时带来的副作用，避免只谈正确率不谈拒识。

## 输出结果设计

脚本应输出两类结果。

### 1. 汇总结果

建议输出 `summary.json`，至少包含：

- 数据集根目录
- 抽样配置
- 实际样本数
- 各子集样本数
- baseline 总体指标
- stabilized 总体指标
- 各子集分项指标

### 2. 逐样本明细

建议输出 `samples.csv`，至少包含：

- `relative_path`
- `subset`
- `gt_full_text`
- `gt_eval_text`
- `baseline_text`
- `stabilized_text`
- `baseline_exact_match`
- `stabilized_exact_match`
- `baseline_char_accuracy`
- `stabilized_char_accuracy`
- `baseline_is_null`
- `stabilized_is_null`

逐样本明细是这次设计中的必要产物，因为后续论文写作和结果解释需要人工抽查失败样本，不能只保留汇总数字。

## 结果解释原则

本次评估允许出现以下结果：

- 优化后整牌完全正确率提升
- 优化后字符级准确率提升但整牌完全正确率变化不大
- 优化后无结果样本数增加，但错误脏输出减少
- 不同子集上的收益不一致

这些结果都应被原样记录，不做超出数据的推断。

推荐的结论风格应是：

- “在本次 CCPD 小样本评估中，优化后 OCR 在若干子集上表现更稳健 / 更保守”
- “是否显著提升，以实际统计结果为准”

不应提前预设“优化后一定更准”。

## 错误处理

- 文件不存在：跳过并记录
- 文件名无法解析：跳过并记录
- 裁牌区域无效：跳过并记录
- 单样本 OCR 失败：该模式记为无结果，并在明细中保留错误标记
- 若某子集可用样本数不足：按实际样本数评估，不中断整次任务

汇总指标的分母应使用“成功进入评估流程的有效样本数”，同时在汇总里额外记录跳过样本数，避免口径含混。

## 实现边界

本次实现建议新增独立评估脚本，而不是改生产推理入口。

建议结构如下：

- 新增脚本：`D:\Projects\Car\scripts\evaluate_ocr_small_sample.py`
- 如有必要，新增实验辅助模块到 `src/car_system/experiments/` 或 `src/car_system/data/`

实现时应注意两点：

1. “优化前 OCR”参考实现应独立存在，避免改坏现有正式 OCR 类
2. 评估逻辑与生产推理逻辑应解耦，避免为了做实验而污染主链

## 测试策略

至少补以下测试：

- CCPD 车牌索引解码测试
- 完整真值到评估真值的规范化测试
- 字符级准确率计算测试
- 分层抽样的可复现性测试
- baseline 与 stabilized 两种模式的结果汇总测试

测试应优先使用假 OCR 或可控桩对象，避免把真实 PaddleOCR 依赖引入单元测试。

## 论文价值

这次补充评估的价值不在于制造一个很大的数字，而在于给 OCR 稳定化提供一份更扎实的“小样本对照证据”。

如果结果显示优化后更准，可以据实写“在小样本上有一定改善”。
如果结果显示优化后只是更保守，也可以据实写“减少了明显脏输出，但准确率提升有限”。

这两种结论都比没有真值支撑更适合写入论文。
