# 车牌专用 PaddleOCR 微调设计

**日期：** 2026-04-13

## 目标

将当前系统中的 OCR 方案从“通用文本识别 + 规则修补”调整为“面向车牌任务的专用 PaddleOCR 识别模型”，优先追求真实识别准确率提升，而不是仅提升输出规整性或保守性。

本次设计只回答一个问题：

在保持现有 `YOLO` 检测主链不变的前提下，是否可以通过构建车牌专用识别数据集并微调 PaddleOCR 识别模型，得到一个比当前通用 OCR 更适合车牌任务的识别后端。

## 范围

### 纳入范围

- 保留现有车辆检测与车牌检测主链
- 基于 `CCPD2019` 构建车牌专用 OCR 训练集
- 使用 PaddleOCR 识别模型作为微调基础
- 训练并评估车牌专用识别模型
- 将新识别模型接回现有 `PaddlePlateOCR` 接口
- 用固定测试集和固定小样本集做前后对比

### 不纳入范围

- 更换车牌检测模型
- 引入 HyperLPR、LPRNet 等第二套主 OCR 路线
- 继续把规则后处理作为主要提升手段
- RK3588、RKNN、NPU、C++、边缘端优化
- 新增复杂的多模型融合推理流程

## 为什么要调整路线

当前 OCR 路线的核心问题不是“候选图不够多”，而是识别器本身仍然是面向通用文本场景的模型。虽然前一轮通过裁剪外扩、多候选图和保守清洗降低了一部分脏输出，但正式小样本评估没有体现出整体准确率提升，反而带来了明显更多的 `null` 结果。

因此，后续 OCR 优化的重点不应继续放在规则修补上，而应放在“让识别模型本身更适合车牌任务”。

## 总体方案

新的 OCR 路线保持现有系统主干不变：

1. `YOLO` 继续负责车牌检测
2. 基于 CCPD 真值构建车牌专用 OCR 数据集
3. 以 PaddleOCR 识别模型为初始化权重进行微调
4. 将新权重接回当前 `PaddlePlateOCR` 接口
5. 用统一评估脚本比较“当前通用 OCR”与“车牌专用 OCR”

这意味着本次调整只动 OCR 数据与识别模型，不打散已经跑通的检测、匹配、可视化和结果导出主链。

## 数据设计

### 数据来源

- 数据集根目录：`D:\plate_project\CCPD2019`
- 划分文件：
  - `D:\plate_project\CCPD2019\splits\train.txt`
  - `D:\plate_project\CCPD2019\splits\val.txt`
  - `D:\plate_project\CCPD2019\splits\test.txt`

### 样本筛选

- 默认使用所有带车牌的 CCPD 子集
- `ccpd_np` 不纳入
- 不只训练困难子集，也不只训练基础子集
- 训练集应覆盖完整车牌场景分布，包括：
  - `base`
  - `blur`
  - `db`
  - `fn`
  - `rotate`
  - `tilt`
  - `weather`
  - `challenge`

若某些子集在当前本地数据中缺失，则按实际可用数据构建。

### 车牌裁剪方式

本次不再优先采用简单轴对齐框裁剪，而是优先利用 CCPD 文件名中的四点顶点信息做透视纠正，得到更规整的车牌图像。

推荐数据流：

1. 解析 CCPD 文件名
2. 提取四点顶点和完整车牌真值
3. 对原图进行透视变换
4. 输出规整化后的车牌小图
5. 为 PaddleOCR 生成识别训练标注

这样做的原因是：

- 车牌本身天然带有长宽比约束
- `rotate` 与 `tilt` 子集中的主要难点之一就是几何畸变
- 透视纠正后的训练样本更符合识别模型的输入假设

### 训练标签

训练标签采用**完整车牌文本**，即保留：

- 省份简称
- 英文字母
- 数字
- 必要的特殊字符，如 `警`、`学`

不再把主训练目标降级为只保留后 6 位或只保留 ASCII 的兼容口径。

同时可以保留一份辅助评估口径：

- `gt_full_text`：完整车牌真值
- `gt_eval_text`：用于兼容旧评估流程的简化文本

其中，`gt_full_text` 是新 OCR 的主训练和主评估目标。

### 字符表

本次不再沿用通用大字典，而是使用车牌专用小字典。

字符范围应来自 CCPD 官方编码表，覆盖：

- 省份简称
- 拉丁字母
- 数字
- `警`
- `学`
- 其他确有需要的车牌特殊字符

使用小字典的目的在于缩小输出空间，减少无关字符干扰，让模型更贴近车牌任务本身。

## 模型与训练设计

### 初始化模型

继续使用 PaddleOCR 识别模型作为基础模型，并以轻量识别模型作为优先初始化来源，例如：

- `PP-OCRv5_mobile_rec`

理由：

- 与当前系统现有 `PaddlePlateOCR` 接口兼容性最好
- 推理端改动最小
- 更适合在当前毕业设计的 PC 端范围内快速落地

### 训练目标

训练目标是得到一个专门面向车牌任务的 PaddleOCR 识别模型，而不是继续提升通用文字识别能力。

训练后模型应具备以下特点：

- 更强的车牌字符模式学习能力
- 更好的完整牌照输出稳定性
- 更少的无关字符干扰
- 在旋转、倾斜、模糊场景下优于当前通用 OCR

### 轻量预处理策略

推理阶段允许保留少量轻量预处理，例如：

- 小幅裁剪外扩
- 必要的透视纠正
- 限量的图像增强

但这些内容不再被视为主要提升来源。主要提升来源应来自专用识别模型本身。

## 评估设计

评估分为两层。

### 1. 识别模型层评估

目标：先证明“车牌专用 OCR 模型本身更准”。

对比对象：

- 当前通用 `PaddlePlateOCR`
- 微调后的车牌专用 PaddleOCR

评估数据：

- 固定使用 `test.txt`
- 保留当前小样本脚本可复现能力
- 先做固定小样本快速评估，再做更大规模验证

评估指标：

- 主指标：整牌完全正确率
- 辅助指标：字符级准确率
- 补充指标：无结果样本数

其中主评估口径优先使用完整车牌文本；兼容旧实验时，可同时保留简化口径。

### 2. 系统接入层评估

目标：证明新 OCR 接回现有系统后仍能稳定运行。

验证内容：

- 单图 smoke
- 短视频 smoke
- JSON/CSV 导出结果是否正常
- 同一批样例上的 OCR 输出是否优于当前版本

系统层评估不追求复杂指标，重点是：

- 能跑通
- 能演示
- 输出质量比当前版本更好

## 成功标准

这次 OCR 重规划要成立，最低成功标准为：

1. 固定测试集上，整牌完全正确率高于当前通用 OCR
2. 字符级准确率同步提升
3. 不能通过大幅增加 `null` 来换取表面规整性
4. 接回现有主链后，图片和视频 smoke 能稳定运行

若只在部分子集上改善，或只减少错误脏输出但不提升准确率，应据实写成局部改进，不夸大为整体成功。

## 系统替换策略

为了降低风险，新的车牌专用 OCR 不应直接覆盖旧 OCR，而应采用渐进替换：

1. 保留当前通用 OCR 作为 baseline
2. 新增一条“车牌专用 OCR”配置入口
3. 小样本与固定测试集对比通过后，再考虑切为默认 OCR

这样做有三个好处：

- 旧链路可以回退
- 论文里更容易写成明确对照实验
- 出现问题时不影响现有演示主链

## 风险与控制

### 风险 1：数据构建质量不稳定

若透视纠正实现不好，训练样本本身可能失真。

控制方式：

- 先对少量样本可视化检查
- 再批量生成

### 风险 2：训练成本超出预期

若直接全量训练，可能拉长周期。

控制方式：

- 先做小规模训练闭环验证可行性
- 再决定是否扩大训练规模

### 风险 3：模型虽在离线评估提升，但接回主链后收益不明显

控制方式：

- 训练内评估先过关
- 再做主链接入

### 风险 4：仍然沿用旧口径，导致新模型价值被低估

控制方式：

- 新模型优先以完整车牌文本为主评估目标
- 旧简化口径仅作为横向兼容指标

## 论文价值

如果该路线成功，论文中 OCR 部分的价值将从“工程稳定化修补”升级为：

- 面向车牌任务的数据构建
- 面向车牌场景的 PaddleOCR 专用化微调
- 基于统一测试集的前后对照验证

这比继续描述规则后处理更容易成立，也更符合“真正提高识别准确率”的目标。

## 本次设计结论

推荐路线为：

**保持 `YOLO + PaddleOCR` 技术路线不变，但将 OCR 从通用识别改为车牌专用识别模型微调。**

该方案兼顾了：

- 准确率提升目标
- 现有系统兼容性
- 工程改动可控性
- 论文表述一致性

因此，这是当前项目在“继续使用 PaddleOCR”前提下，最合理的 OCR 重规划方向。
