# 第3章 系统设计与实现

## 3.1 系统总体设计

本文面向 PC 端场景，设计并实现了一个基于深度学习的车辆类型与车牌检测识别系统。系统的核心目标不是面向边缘端部署优化，而是在个人计算机和远端训练环境下完成完整的检测、识别、可视化与实验验证流程，从而满足毕业设计中“能运行、能演示、能写论文”的要求。

从整体流程上看，系统采用“双检测器 + OCR”的技术路线。首先，车辆检测器负责从输入图像或视频帧中定位车辆目标，并给出 `car`、`bus`、`truck`、`motorcycle` 等粗粒度类别；随后，车牌检测器对同一图像中的车牌区域进行检测；在得到车牌位置后，系统对车牌区域进行裁剪，并交由 OCR 模块识别车牌字符；最后，将识别结果与对应车辆目标进行匹配，并输出标注图像、标注视频、JSON 结果和 CSV 结果。

该方案的优点在于模块边界清晰、训练和调试难度较低，既便于工程实现，也便于论文描述。与“单模型联合完成所有任务”的方式相比，双模型方案更适合当前毕业设计的开发周期和实验条件。

## 3.2 系统架构设计

系统从实现层面可划分为配置模块、检测模块、OCR 模块、流水线模块、输入输出模块、可视化演示模块和实验评估模块。各模块之间通过明确的数据结构和函数接口协同工作，共同组成完整的端到端识别流程。

配置模块负责统一管理系统运行参数，包括车辆检测权重路径、车牌检测权重路径、OCR 运行模式、专用 OCR 模型路径、字符字典路径以及结果输出目录等。通过将这些参数集中写入 YAML 配置文件，系统能够在不修改源码的前提下切换不同模型和运行策略，提高了实验复现和演示部署的便利性。

检测模块以 YOLO 接口作为统一后端，分别实例化车辆检测器和车牌检测器。车辆检测器用于识别车辆类型并输出边界框，车牌检测器用于识别车牌位置并输出车牌边界框。当前最终方案中，车牌检测权重采用基于 CCPD 扩展数据集训练得到的 mvp 模型，该模型在后续实验中表现优于 quick 基线。

OCR 模块最终采用 PaddleOCR 框架，并基于 CCPD 裁剪车牌数据对 `PP-OCRv5_mobile_rec` 模型进行了车牌任务微调。系统在运行时支持 `generic` 与 `specialized` 两种模式，其中 `generic` 用于通用 OCR baseline 对比，`specialized` 作为最终主识别方案。这样既保留了基线比较条件，也使系统能够在最终演示阶段直接切换到专用识别模型。

流水线模块负责组织整体推理过程，是连接各模块的核心。它完成车辆检测、车牌检测、目标匹配、车牌裁剪、OCR 识别和结果封装等操作，并将原始模型输出统一转换为最终业务结果。图片推理、视频推理与前端演示界面均复用了这一核心流水线逻辑。

输入输出模块负责图像读取、视频逐帧读取、标注图保存、标注视频保存、JSON 导出和 CSV 导出。该模块保证了系统既可以用于静态图像测试，也可以用于动态视频演示和后续实验统计。

可视化演示模块基于 Streamlit 实现，用于构建轻量级的 PC 端交互界面。用户可以上传图片或视频，执行推理，并查看标注结果与结构化表格。对于毕业设计场景而言，这一模块的重点在于稳定展示完整流程，而不是追求复杂的前端交互设计。

实验评估模块用于支撑论文中的实验整理和结果分析。项目中已实现检测训练结果汇总、OCR 小样本评估、专用 OCR 离线评估等脚本，能够为论文中的对比实验和定量分析提供数据支撑。

## 3.3 核心功能模块实现

### 3.3.1 配置与运行时构建

系统首先通过配置模块读取 YAML 文件，并解析出检测器、OCR 与输出相关参数。运行时构建阶段根据配置内容实例化车辆检测器、车牌检测器和 OCR 引擎。当前实现中，检测器还支持单独指定 `device` 参数，因此可以根据运行环境选择 CPU 或 GPU 推理策略。例如，在最终远端演示配置中，由于容器环境下 Ultralytics 默认 GPU 推理存在兼容问题，系统将检测器固定为 CPU，以保证演示链路稳定运行。

### 3.3.2 车辆与车牌检测实现

检测模块统一封装了 YOLO 模型的加载与预测逻辑。车辆检测器和车牌检测器在接口层保持一致，均输出目标类别、置信度和边界框信息。这样一来，后续流水线无需关心具体模型来源，只需要处理统一格式的检测结果即可。

在本课题中，车辆检测主要承担“车辆类型粗识别”的功能，而车牌检测承担“精确定位车牌区域”的功能。通过将这两个任务拆分为两个独立 detector，不仅降低了数据准备和模型训练难度，也使得系统在调试与实验阶段更易于定位问题来源。

### 3.3.3 OCR 识别实现

OCR 模块在最终版本中采用专用 PaddleOCR 识别模型作为主方案。系统在车牌裁剪后将图像送入 OCR 引擎，并输出识别文本与置信度。为了提高演示阶段的稳定性，OCR 推理前仍保留了小幅裁剪外扩、少量候选图像构造和轻量文本归一化等工程处理，但这些规则不再作为准确率提升的核心来源。

在项目中，OCR 部分经历了两阶段演化。第一阶段主要通过裁剪外扩、多候选图像和保守清洗来提升输出规整性；第二阶段则转向基于 CCPD 构建车牌专用识别数据集，并微调 PaddleOCR 识别模型。后续实验结果表明，真正带来识别准确率提升的是第二阶段的专用识别模型，因此本文最终采用 specialized OCR 作为系统主识别方案。

### 3.3.4 目标匹配与结果组织

在检测器分别输出车辆框和车牌框后，流水线模块根据空间位置关系完成车牌与车辆的匹配。匹配完成后，系统将每个车牌识别结果与对应车辆目标绑定，并最终组织为统一的结果结构。该结构同时服务于可视化绘制、JSON 序列化、CSV 导出和前端表格展示。

这种设计的优势在于，后续模块不需要再次关心底层模型输出细节，而只需围绕统一的中间结果进行处理。这提高了系统整体的可维护性，也方便了论文中对输出结果形式的说明。

### 3.3.5 图像与视频处理实现

系统同时支持图片推理和视频推理。对于图片输入，流水线只需执行一次检测与识别并生成标注图像；对于视频输入，系统则逐帧执行同样的检测、匹配与识别操作，并最终将标注帧重新编码为输出视频。与此同时，系统会同步导出 JSON 和 CSV 文件，用于保存结构化实验结果。

这一设计使系统不仅能够满足演示需求，也能够支持后续实验数据统计与论文素材整理。在毕业设计场景中，这种“推理结果 + 可视化输出 + 结构化导出”的完整链路具有较高的实用价值。

## 3.4 系统运行流程

系统的运行流程可以概括为以下几个步骤：首先，用户通过命令行脚本或 Streamlit 界面输入图片或视频；其次，系统加载配置文件并构建运行时对象，包括车辆检测器、车牌检测器和 OCR 引擎；然后，对输入图像或视频帧执行车辆检测和车牌检测；在此基础上，系统完成车牌与车辆的匹配，并对车牌区域进行裁剪；接着，将裁剪后的车牌图像送入 OCR 模块进行识别；最后，将检测结果、识别结果和匹配关系统一组织并导出为标注图像、标注视频、JSON 和 CSV 文件。

从实现角度看，系统的图片处理和视频处理共享同一套核心推理逻辑，因此整体结构较为统一。图片路径和视频路径之间的差别主要体现在输入读取方式和最终输出形式上，而核心检测与识别流程保持一致。

## 3.5 本章小结

本章从系统总体设计、架构划分、核心模块实现和运行流程四个方面，对基于深度学习的车辆类型与车牌检测识别系统进行了说明。系统采用双检测器加 OCR 的实现路线，模块边界清晰，既便于工程实现，也便于后续实验验证与论文撰写。

在最终实现中，车牌检测部分采用基于扩展数据集训练得到的 mvp 模型，OCR 部分采用基于 CCPD 车牌裁剪数据微调得到的专用 PaddleOCR 模型。系统已经完成图片与视频场景下的完整联调，能够输出可视化结果与结构化结果文件，为下一章的实验分析和性能评价奠定了实现基础。
