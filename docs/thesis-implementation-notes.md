# Thesis Implementation Notes

## 系统实现概述

本系统采用“双检测器 + OCR”的总体架构。车辆检测器负责输出 `car`、`bus`、`truck`、`motorcycle` 等粗粒度车辆类别；车牌检测器负责定位车牌区域；随后系统从原图中裁剪车牌区域，交由 PaddleOCR 识别字符，并将识别结果与对应车辆目标进行匹配，最终导出可视化结果、JSON 结果和 CSV 结果。需要说明的是，题目中的“车辆类型”在当前实现中主要表现为系统级粗粒度类别输出，而本文真正开展量化优化与实验验证的重点是车牌检测和车牌 OCR 两个环节。

与早期“通用 OCR + 规则清洗”的实现不同，当前最终方案已经切换为“YOLO 检测 + 车牌专用 PaddleOCR 识别”。也就是说，OCR 部分不再把基于规则的稳定化作为最终提升准确率的核心方案，而是将其保留为轻量辅助处理，真正的识别增益主要来自车牌专用识别模型的微调和接入。

## 主要功能模块

### 1. 配置模块

配置模块负责读取 YAML 运行参数，包括车辆检测权重、车牌检测权重、OCR 模式、专用 OCR 模型路径、字符字典路径以及输出保存选项。当前配置系统已经支持为 detector 单独指定 `device`，因此可以根据运行环境在不改源码的前提下切换 CPU 或 GPU 推理策略。

### 2. 检测模块

检测模块以 YOLO 接口作为统一后端，车辆检测与车牌检测分别实例化为两个独立 detector 对象。这种方式降低了工程耦合度，也更符合本课题“先完成可运行系统，再进行实验验证”的开发目标。最终车牌检测主权重使用 `ccpd_yolo26n_mvp` 方案，quick 方案保留为对比基线。

### 3. OCR 模块

OCR 模块最终采用 PaddleOCR 识别框架，并基于 CCPD 裁剪车牌图像构建的识别数据集，对 `PP-OCRv5_mobile_rec` 进行了车牌任务微调。系统在运行时支持两种模式：

- `generic`：通用 PaddleOCR 识别器，用作 baseline。
- `specialized`：车牌专用 PaddleOCR 识别器，用作最终主识别方案。

在 OCR 推理前，系统仍保留了小幅裁剪外扩、少量候选图像构造和轻量文本归一化等工程处理，但这些处理不再作为“准确率提升”的主要论据。早期的 `50` 样本小评估已经说明，仅靠规则稳定化并不能稳定提高整体识别精度；后续真正带来改善的是专用识别模型本身。

### 4. 流水线模块

流水线模块负责组织车辆检测、车牌检测、目标匹配、车牌裁剪、OCR 识别与结果封装，是整个系统的核心。它将原始模型输出转换为最终业务结果，并统一供图片推理、视频推理与演示界面复用。

### 5. 输入输出模块

I/O 模块负责图像读取、视频逐帧读取、标注图保存、标注视频保存、JSON 导出和 CSV 导出。该模块保证了系统既能用于单张样例验证，也能用于视频演示和实验结果统计。

### 6. 可视化与演示模块

系统提供 PC 端演示能力，能够完成图片或视频输入、推理执行、结果展示以及结构化输出查看。对于毕业设计场景而言，这部分实现的重点是“稳定展示完整流程”，而不是追求复杂的部署形态。

### 7. 实验脚本与评估模块

项目中已实现检测训练汇总、OCR 小样本评估、专用 OCR 训练包装、专用 OCR 离线评估等脚本，可支撑论文中的训练结果整理、对比实验和结果复核。

## 实现特点

- 全项目采用 Python 实现，开发效率高，代码级验证与脚本级复核成本较低。
- 检测、OCR、I/O、评估等模块职责清晰，便于单独测试和论文描述。
- 技术路线聚焦于 PC 端系统实现与实验验证，没有扩展到 RK3588、RKNN、NPU 或 C++ 重构等超出课题边界的内容。
- OCR 部分经历了从“规则稳定化”到“专用识别模型”的明确路线收敛，最终实现与“真正提高识别准确率”的目标保持一致。

## 复现与运行说明

当前项目仓库已经保留了系统主链代码、配置模板、评估脚本和测试用例，能够支持本地代码级验证、脚本级联调和论文数据复核。以当前环境为例，`python -m pytest -q` 已可通过完整测试集合，这说明系统的软件结构和主要逻辑处于可验证状态。

但需要明确区分“代码可验证”与“最终演示可完整复现”这两个层次。当前本地仓库中的 `plate_ocr_specialized.local.yaml` 主要作为本地专用 OCR 配置模板使用，而最终演示版本使用的是远端 `remote_mvp_specialized.yaml` 配置。该远端配置直接引用了服务器上的检测权重、专用 OCR inference 模型和字符字典路径，因此与当前本地工作区并不是完全等价的运行环境。

换言之，当前仓库可以完成以下工作：代码阅读与修改、测试验证、检测和 OCR 评估脚本运行、论文结果整理与复核。而如果要在本地完整复现最终演示效果，仍需额外补齐车辆检测权重、车牌检测权重以及车牌专用 PaddleOCR 导出模型。本文中“最终系统联调通过”的表述，特指在远端训练环境和远端最终配置下完成的真实验证结果。

## 当前工程状态

截至目前，系统的软件框架、输入输出链路、结果序列化、检测训练脚本、OCR 评估脚本和远端运行环境均已完成。车辆检测与车牌检测主链已经能够稳定完成图片和视频推理，检测结果可视化以及 JSON/CSV 导出也已打通。

车牌检测部分，扩展数据集训练得到的 mvp 权重在 `mAP@0.5:0.95` 上达到 `0.78672`，优于 quick 基线的 `0.77963`。因此，最终检测主权重已经固定为 mvp 方案。

OCR 部分，早期基于规则的稳定化策略在 `50` 张 CCPD 小样本评估中未体现出整体精度提升，因此未被保留为最终主要方案。后续重新构建了 CCPD 识别训练集，并完成了 PaddleOCR 专用识别模型的训练、导出与评估。训练数据规模为 `train 20000 / val 5000 / test 10000`，训练轮数为 `75`，在第 `74` 个 epoch 取得最优验证指标 `acc = 0.9856`。

在完整 `10000` 张测试样本上的真实对比结果中，通用 `PP-OCRv5_mobile_rec` 的整牌完全正确率为 `2.52%`，字符级准确率为 `31.32%`；车牌专用 PaddleOCR 的整牌完全正确率提升到 `14.80%`，字符级准确率提升到 `38.28%`。与此同时，无结果比例从 `28.20%` 上升到 `32.79%`，说明专用 OCR 在提升准确率的同时仍保留一定保守输出倾向。总体上，这一结果已经足以支持“专用 OCR 优于通用 OCR”的结论。

## 最终联调状态

当前远端最终演示链路已经接入专用 OCR，配置文件为：

```text
/root/autodl-tmp/car-project/configs/remote_mvp_specialized.yaml
```

在这版配置中，OCR 模式为 `specialized`，识别模型直接加载导出的专用 inference 模型。由于远端环境中 Ultralytics 默认 GPU 推理在当前容器下存在运行时兼容问题，最终演示配置将车辆检测器与车牌检测器固定为 `cpu`，以保证主链 smoke 演示稳定完成。该调整属于工程运行策略，不改变本文对检测模型和 OCR 模型本身效果的实验结论。

基于该最终配置，远端图片 smoke 与视频 smoke 均已通过，说明“检测 + 专用 OCR + 可视化输出”的完整系统主链已经跑通。当前剩余工作重点已不在继续改算法，而在于将这些真实结果整理为论文正文、图表和答辩材料。
